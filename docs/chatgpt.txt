# Executive Summary

We recommend using **Bevy 0.19** for the engine (with wgpu for rendering) and compiling to WebAssembly (WASM) for the browser client. WASM builds are single-threaded (threads disabled by default) and can be large (20–30 MB in examples), so size optimization (LTO, `-Oz`, etc.) is critical. For in-game scripting, **Rhai** is a suitable choice: it is an embedded AST-based interpreter (≈2× slower than Python) that is “fast enough for most real-life scenarios”. A Bevy scripting plugin (e.g. `bevy_mod_scripting`) supports Rhai but currently does *not* compile to WASM, so we may embed Rhai directly or use a custom plugin for the WASM client.

For networking, we suggest using WebRTC DataChannels for peer-to-peer (Mode 2) communication. The [matchbox](https://github.com/johanhelsing/matchbox) crates (e.g. **bevy_matchbox**, **matchbox_socket**, **matchbox_server**) provide unified support for WASM and native, handling signaling and allowing both unreliable/unordered and reliable data channels. For signaling, a simple WebSocket-based server can exchange SDP/ICE messages (bevy_matchbox automates this). NAT traversal will require STUN (success ~75–80% of sessions) and TURN for the remainder; in symmetric-NAT environments STUN alone succeeds only ~20–40% of the time, implying a high reliance on TURN relays. We plan two channels per peer pair: an *unreliable unordered* channel for frequent state updates and a *reliable* channel for game events, as matchbox supports both.

State replication depends on mode: for Mode 3 (server-authoritative), **bevy_replicon** can serialize and sync the world from server to clients (it is explicitly designed for authoritative replication). For Mode 2 (full P2P), no off-the-shelf library fully handles multi-authority; we may use **Lightyear** (active dev, supports client-prediction and flexible replication) or **Aeronet** (modular IO/transport layers) with custom logic, but likely will build some bespoke synchronization. In practice, we’ll reuse existing crates: e.g. use replicon (or aeronet_replicon) on the server, and use low-level messaging (matchbox/renet) on peers. We advise against runtime CRDTs due to complexity. 

Anti-cheat: In Mode 3, the server is authoritative, inherently preventing client-side tampering. In P2P (Mode 2), each peer must perform “plausibility checks” (e.g. velocity/position bounds) on received updates and could sign outgoing messages to ensure authenticity, but these only provide tamper *evidence*, not prevention. Without central authority, cheating cannot be fully prevented (only minimized by trust assumptions).  

For hosting (Mode 3 servers and services), containerized deployment on cloud (e.g. Kubernetes/EKS) is recommended for scalability. Dedicated bare-metal servers yield more consistent performance per dollar but require manual provisioning. Note that global scaling can greatly raise costs: AWS GameLift estimates for 8 regions were 279% higher than a single region. Container orchestration (EKS, ECS) offers flexibility and lower DevOps overhead compared to managed fleets. 

**Implications:** We prioritize using existing game-focused Rust/WASM libraries (Bevy, matchbox, replicon) where possible. Key challenges include managing WASM bundle size and ensuring robust NAT traversal (use TURN for ~20–40% of cases). We will prototype early to benchmark WASM size and performance. Risks include stagnation of any chosen library (monitor crate health) and the inherent complexity of peer-to-peer cheat mitigation. 

**Next Steps:** Finalize crate versions (e.g. Bevy v0.19, latest matchbox, Rhai v1.x), implement a minimal proof-of-concept (Hello World over matchbox with Rhai script), measure binary sizes, and draft architecture diagrams (signaling flow, server/client components). Setup benchmarks for WASM download size, tick rates, and simulate mode-2 mesh loads. Build a risk register with mitigation plans (e.g. fallback to server-only mode if P2P fails).

## Scope and Methodology

We addressed each subsystem in the prompt (A–G) by searching recent sources (2022–2026) on Rust/WASM game development, networking, and cloud hosting. We prioritized official docs and credible posts (Bevy docs, Rhai site, AWS blogs, etc.). Key questions included: *What is current WASM support in Bevy? How large are binaries?* (*Findings:* tens of MB unless heavily optimized.) *How to embed Rhai in WASM?* (*Findings:* Rhai itself compiles to WASM; plugins like `bevy_mod_scripting` support Rhai but not WASM yet.) *Which Rust crates for WebRTC exist?* (*Findings:* `webrtc-rs` and `str0m` are core implementations; `matchbox` wraps them for game use.) *How to handle NAT?* (*Findings:* Use STUN + TURN; STUN alone fails often, especially under symmetric NAT.*) *Which networking crate for Bevy?* (Lightyear, aeronet, replicon, renet). We synthesized their capabilities from docs and repos. For hosting, we examined AWS game server blogs and cost analyses.

## A. Engine & Rendering (Bevy + WASM)

- **Bevy Version:** Use the latest stable (v0.19 as of mid-2026) since it has matured significantly. Bevy’s WASM support still runs single-threaded in browsers (WASM threads disabled by default).
- **WASM Binary Size:** Unoptimized Bevy WASM builds are large (on-disk sizes on the order of 20–30+ MB). In one example, a simple 3.7 MB compressed WASM (8 MB); more complex demos hit ~24–33 MB uncompressed. We must heavily optimize (e.g. `wasm-opt`, LTO, `-Oz`, remove unused features) and serve with gzip/Brotli to reduce transfer to a few MB.
- **Fixed Timestep:** Bevy supports custom fixed timesteps (use `FixedTimestep` schedule). For a headless server (Mode 3), use a minimal Bevy configuration (no graphics subsystem) or the [bevy-headless](https://crates.io/crates/bevy_headless) bundle to reduce overhead. No specialized crate needed beyond setting `--release --features bevy/headless`.
- **Implication:** Expect 10–30 MB of initial payload in WASM. Aim for ~5–10 MB transfers after compression. Frontend must show a loading indicator or use progressive loading. Enforce single-thread logic in code (no `bevy::tasks`, use cooperative scheduling). 

## B. Scripting Runtime (Rhai)

- **Rhai Features:** Rhai is an embedded scripting language with an AST interpreter (no JIT/bytecode). It is designed for ease of embedding. Its benchmarks show it is roughly 2× slower than Python on typical tasks, which is acceptable for game logic/control scripts.
- **Integration with Bevy:** The `bevy_mod_scripting` plugin supports Rhai (among others) for scripting and is under active development. However, it currently *does not compile to WASM* (see its README: “Supported Platforms: WASM no, see issue #119”). Alternatives: embed Rhai manually (use `rhai = "x.x"` crate in WASM target) and call it in systems, or write a small custom plugin. 
- **Sandboxing & Limits:** Rhai allows limiting script execution via `Engine::set_max_operations` (prevent infinite loops) and has configurable features (disable I/O, etc.). By default, Rhai does not impose time/CPU limits, so we must set these when running untrusted code. 
- **Data Types:** Rhai supports basic types (numbers, strings, arrays, maps) and can register custom host functions (like small-vector math). For game logic (UI, spawning, simple physics), these should suffice. Heavy lifting (e.g. physics, asset loading) stays in Rust code.
- **Memory/Performance:** Rhai has its own memory management (it’s safe for WASM) but large scripts or frequent execution could add overhead. Keep scripts small and use native Rust for performance-critical loops. 
- **Implication:** Rhai is a workable choice for in-game logic, especially as it’s pure Rust. Expect minor slowdown; test on target hardware. Ensure to set operation limits for safety. 

## C. Networking Transport (WebRTC DataChannels)

- **WebRTC Libraries (Rust):** The main Rust WebRTC stacks are:
  - [webrtc-rs](https://crates.io/crates/webrtc) (also called `webrtc` crate) — a full implementation of the WebRTC spec (DataChannels, ICE, etc.).
  - [str0m](https://github.com/str0mio/webrtc-rs) — a “sans-IO” lower-level WebRTC library (C++ based) for DataChannels/SCTP (v0.1.0 in 2022).
  - For our purposes, we may not call these directly; instead, we use higher-level libraries.
- **Matchbox (Bevy P2P):** The `matchbox` project (Crates: `matchbox_socket`, `matchbox_server`, `bevy_matchbox`) wraps WebRTC for games. It provides:
  - Signal server (`matchbox_server`) for WebSocket signaling.
  - A unified socket API for WASM/native using the same data channel logic.
  - Supports **both unreliable/unordered and reliable channels** (configurable).
- **Usage:** With matchbox, client code needs only to connect to a signaling URL and then process a future to handle incoming events. The plugin hides SDP and ICE exchange (see matchbox docs). For example, using `bevy_matchbox`, just give it a URL (ws://) and it handles offer/answer and ICE under the hood.
- **Channel Configuration:** We will open two channels per peer pair (if matchbox allows multiple): one set as `ordered=false, maxRetransmits=0` for high-rate updates (UDP-like), and another `ordered=true, reliable` for low-rate events (e.g. spawn/despawn). `matchbox_socket::Config` supports this.
- **NAT Traversal:** Expect ~75–80% success with STUN-only WebRTC (server-reflexive candidates). The rest (~20–25%) require relay (TURN). Symmetric NATs (common on cellular) succeed only ~20–40% with STUN (meaning ~60–80% would need TURN). Plan to provide a TURN server or use a public one to cover difficult cases.
- **Signaling Protocol:** We will implement a simple JSON-over-WebSocket exchange (typical pattern). The signaling server (matchbox_server) uses a pub-sub mesh by default, but we could adapt it. The focus is P2P, so no central gameplay logic passes through it except SDP/ICE.
- **Implication:** WebRTC DataChannels are suitable for realtime game data. Using matchbox simplifies integration. Our architecture will include a signaling service (could be a small BE server). Unavoidable heavy design: rely on TURN (increases server cost) for up to ~25% of sessions.

## D. State Replication (Game State Sync)

- **Mode 1 (Standalone):** No networking; full authority locally.
- **Mode 2 (P2P Full-Mesh):** Each client is authoritative over *its own player state* (inputs/position). Other entities’ states come from their owners. This requires merging multiple state streams. No existing crate directly solves this for Bevy. We plan:
  - On each peer, simulate local player and possibly all entities (client-side prediction).
  - Send local state updates on the unreliable channel.
  - Receive others’ state updates; apply interpolation/prediction.
  - Implement simple anti-cheat: bound checks, reject physically impossible states.
- **Mode 3 (Server-Authoritative):** The server is the sole authority. Clients send input commands (e.g. “move”) to server reliably; server updates world and sends authoritative state back.
  - For this, use **bevy_replicon**: “a server-authoritative replication crate for Bevy”. It can serialize component changes and spawn/despawn entities. We would pair this with a network layer (e.g. aeronet_replicon or custom using matchbox/renet).
  - Alternatively, **Lightyear**: designed for client-server multiplayer with prediction/interpolation. Lightyear is newer and provides higher-level sync support (entity, component replication). It also has a plugin to integrate with Bevy ECS.
  - **Aeronet**: A lower-level “Bevy-native networking library” focusing on reliable packet delivery. It suggests using aeronet_replicon for replication. Aeronet v0.21 is stable for Bevy 0.19.
- **Build vs Reuse:** We recommend reusing `bevy_replicon` or Lightyear where possible rather than building a custom protocol. Evidence: bevy_replicon already handles authoritative scenarios. Lightyear has broad features (prediction, rollback). If neither fully meets the P2P case, we could use their building blocks (e.g. networking, serialization) but implement the multi-authority logic ourselves.
- **Networking Layer:** For Mode 3, use a reliable transport (e.g. matchbox’s reliable channel or renet over UDP/WSS). For Mode 2, use the unreliable channel as stated.
- **Implication:** We’ll likely adopt a hybrid: bevy_replicon on the server side, and custom handling on clients. Maintain a consistent tick rate (e.g. 20–60 Hz). We will benchmark tick performance (see Benchmarks section).
  
## E. Anti-Cheat & Verification

- **Mode 3 (Server):** Server enforces all rules, so clients cannot cheat by altering game state (only allowed to send inputs).
- **Mode 2 (P2P):** Each client’s local state is authoritative for itself. Other clients can at best perform “plausibility checks”:
  - Eg. ensure incoming position changes are within max velocity/time bounds.
  - Limit unrealistic actions (teleport suddenly).
  - However, any client can still send arbitrary (cheating) moves; checks only flag obvious bad data. 
- **Digital Signatures:** We could sign client messages (with a public key infrastructure or a shared secret) to ensure the message came from a genuine client. This prevents simple impersonation/spoofing but *does not stop the client from lying about position*. It does give tamper evidence.
- **Frequent Snapshotting:** Sending state snapshots (with previous states) helps clients detect missing data, but doesn’t inherently prevent cheating. 
- **Peer Trust:** In P2P, one could introduce a credibility/reputation system, but complexity is high.
- **Conclusion:** We accept that Mode 2 is inherently vulnerable. We will implement basic consistency checks on each peer to mitigate gross cheats (e.g. no-clipping through obstacles by noticing out-of-bounds coordinates). Full anti-cheat (like Raptr) is beyond scope, so we rely on eventual Mode 3 or out-of-band penalties. 

## F. Deployment & Cloud Services

- **Container vs Bare-Metal:** For mode 3 servers and services (signaling, potential TURN), using containers (e.g. Docker on AWS EKS/ECS) is advantageous for scaling and deployment automation. Kubernetes or similar allows dynamic scaling based on load.
  - *Example:* AWS blog notes that containers improve **compute density** and performance consistency.
- **Cost Considerations:** Cloud providers charge per region. A single-region estimate may look low ($1,330/mo for one region) but supporting multiple regions increases cost by ~279% (to ~$3,713/mo for 8 regions in a typical scenario). This highlights the need for careful budgeting or using global orchestration. 
  - Container orchestration is typically more flexible and lower operational overhead than a proprietary solution like GameLift.
  - Bare-metal or fixed servers eliminate per-instance and bandwidth egress fees; good for predictable usage.
- **NAT Punching Services:** For peer invites, a TURN (relay) service is needed (could self-host [coturn](https://github.com/coturn/coturn)). No external “invite” service is needed beyond our signaling server.
- **Third-Party Services:** We may use:
  - Auth servers or databases (if any central account system is needed).
  - Cloud storage (S3-like) for static assets if any.
  - Matchmaking can be as simple as listing available game sessions on a central server (which then provides signaling URLs).
- **Implication:** We will containerize servers (e.g. build Rust service images) and likely deploy on AWS/GCP/Azure. Planning regions and bandwidth is important to avoid the high multi-region cost bump. The photo above shows typical server racks for reference:

### Benchmarks & Measurements

- **WASM Binary Size:** Based on community reports, expect uncompressed sizes ~20–30 MB for a non-trivial Bevy game. After `wasm-opt` and gzip/Brotli, transfers can drop to ~5–10 MB. *Action:* Benchmark our actual game builds.
- **Frame/Tick Rate:** Plan 30–60 Hz tick. Lightyear/Aeronet claim to support hundreds of ticks (need testing). If simulation is heavy, consider server tick=30 Hz and client interpolation for smoothness.
- **Peer Count (Mode 2):** Full-mesh scales as O(n²) connections. For large groups (8+ players), P2P becomes heavy. We may limit Mode 2 to small lobbies (4–8 players) to keep message count manageable.
- **Latency:** WebRTC latency (peer-to-peer) is typically low (<50ms) on good networks. We’ll measure round-trip ping via DataChannel to estimate.
  
### Risks and Uncertainties

- **Library Stability:** Some crates (bevy_mod_scripting, Lightyear) are under active development. If their APIs change, we may need to maintain patches. Mitigation: Pin crate versions and isolate usage.
- **WASM Performance:** Single-threaded physics/logic might bottleneck on browser. We must test on target (mobile vs desktop) and possibly reduce load or simplify simulation.
- **Networking Complexity:** P2P mesh is complex (NAT, reliability). Fallback: if peer-to-peer fails, use Mode 3 (all players connect through server).
- **Cheating:** P2P mode has inherent cheat risk. We must weigh if a purely P2P mode is acceptable or if we enforce server mode in competitive contexts.
- **Cost Overruns:** Cloud hosting costs can be higher than expected (especially global scale). We will model expected usage with tools (Edgegap calculator or AWS calculator) to budget. Consider hybrid models (peer-hosted sessions).
- **Client-Server Divergence:** Differences in time or float precision (WASM vs native) could cause desync. Use deterministic simulation steps and maybe lockstep for critical logic.
