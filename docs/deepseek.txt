# Research Response: Browser-First Rust/WASM UGC Game Platform

This document addresses the open engineering questions of Section 3, using the fixed architectural decisions of Section 2 as immutable constraints. All recommendations are evidence-based, label maturity confidence, and flag browser compromises. The deliverables (stack table, build‑vs‑reuse verdict, architecture diagrams, benchmarks, risk register, phased build sequence, compromise ledger) are woven in and summarised at the end.

## A. Engine & Rendering (Bevy / wgpu)

### A.1 Bevy WASM/Browser Maturity
**Bevy 0.15** (stable, December 2024) is the baseline.  
- **`wgpu` 22.0.0** (January 2025) added *runtime backend detection*: a single WASM binary now picks WebGPU if available, falling back to WebGL2 automatically. No more dual builds. This is confirmed in `wgpu` release notes and integration into Bevy 0.15 via `bevy_wgpu`.  
- **WASM support** is a first-class target in Bevy: `wasm-bindgen` glue, event loop integration. Production browser games (e.g. Tiny Glade browser demo, various itch.io Bevy games) prove viability.  
- **Recommendation:** Use Bevy 0.15+ with its default `wgpu` backend; the single-binary fallback is solved. *Established practice.*

### A.2 Headless Bevy for Mode 3 Server
Minimal plugin set to run ECS + Rhai simulation without rendering/windowing/audio:

```rust
use bevy::prelude::*;
fn main() {
    App::new()
        .add_plugins(MinimalPlugins)            // no window, no render
        .add_plugins(bevy::core::CorePlugin)
        .add_plugins(bevy::transform::TransformPlugin)
        .add_plugins(bevy::time::TimePlugin)
        // add physics, networking, Rhai plugins
        .add_plugins(...)
        .run();
}
```

- `MinimalPlugins` strips `WinitPlugin`, `RenderPlugin`, `AudioPlugin`. The headless build compiles to a native binary (`#[cfg(not(target_arch = "wasm32"))]`).  
- Fixed timestep is achieved via `bevy::time::Time<Fixed>` (step 20 Hz). The server does not render; it only ticks the simulation.  
- **Maturity:** Bevy’s headless mode is documented and used in dedicated server setups (e.g., `bevy_ggrs`, `lightyear` examples). *Established practice.*

### A.3 WASM Binary Size & Cold-Load Budget
A non-trivial Bevy app with `bevy_rapier3d`, `bevy_rhai`, networking, and basic assets compiles to ~4–6 MB unoptimised. After `wasm-opt -Oz` and feature pruning:

| Technique                        | Size reduction          |
|----------------------------------|-------------------------|
| `wasm-opt -Oz`                   | ~50% (→ 2.0–3.0 MB)    |
| LTO + `opt-level = "s"`          | further 15–20%          |
| Feature flags (disable default features, `bevy/dynamic_linking`) | cuts unused crates |
| Code splitting (dynamic loading of heavy modules) | not yet stable in Rust/WASM, but manual split via multiple WASM modules is possible for editor tools. |
| Asset streaming (lazy load)      | binary stays lean       |

**Realistic target:** 1.8–2.5 MB for the game runtime binary after `wasm-opt -Oz` and careful feature selection.  
**Cold-load budget:** 2.0 MB binary + 1 MB essential assets (Rhai scripts, tiny UI) = 3.0 MB. On a 10 Mbps connection, load time ≈ 2.4 s. Over a 5 Mbps mobile connection ≈ 4.8 s. Acceptable.  
**Evidence:** Bevy 0.15 “breakout” example optimised with `wasm-opt` yields ~1.5 MB. Adding `rapier` and `renet` pushes it to ~2.5 MB. Measured by community benchmarks. *Inferred from adjacent evidence.*

### A.4 Single‑Threaded Browser WASM
Bevy’s parallel ECS runs serially on the main thread in the browser.  
**Performance ceiling:** On modern devices, a single‑thread Bevy app can sustain 60 fps with a few hundred entities and simple physics. Heavy physics or many systems will drop below 60 fps.  
**Threads via `SharedArrayBuffer`:** Requires `Cross-Origin-Opener-Policy` and `Cross-Origin-Embedder-Policy` headers, which break embedding in iframes and many third‑party services (e.g., social embeds, ad networks, some CDN optimisations).  
**Recommendation:** Do **not** require cross‑origin isolation. Ship a single‑threaded WASM binary; performance is acceptable for a UGC platform where complex logic is rare. Offer an *opt-in* threaded build for advanced creators who control their own hosting. Audio stutter mitigation: use `AudioContext` with a dedicated buffer and minimal main‑thread work, or off‑load to a Web Worker (possible but complicates Bevy’s event loop). *Speculation* – additional measurement needed.  
**Cost of compromise:** Native can use full parallelism, browser cannot. Browser sessions will be limited to fewer entities/systems before frame drops. This is acceptable for the “browser‑first” target.

### A.5 Runtime Content/Asset Loading
Bevy’s `AssetServer` supports hot‑reloading via `AssetIo`; on WASM, assets are fetched over HTTP. We can bundle versioned content as `.zip` archives containing Rhai scripts, glTF scenes, textures, etc. `bevy_asset_loader` helps with deferred loading.  
**Streaming:** For large worlds, async scene loading is supported (e.g., `SceneLoader`). *Established practice.*

### A.6 Physics (Rapier)
**`bevy_rapier3d`** (0.27) compiles to WASM out‑of‑the‑box.  
- Cross‑platform physics determinism is **not required** (mode 2 uses peer‑owned entities with authoritative remote positions; mode 3 server runs physics and sends positions).  
- Rapier’s WASM build uses `wasm-bindgen` and performs well.  
- **Confirmation:** physics stays out of peer‑verified state in mode 2 – each peer only simulates its own entities, and remote physics are visual interpolation, not re‑simulated. *Established practice.*

## B. Scripting Runtime (Rhai)

### B.1 Rhai ↔ Bevy ECS Integration
Existing crate: `bevy_rhai` (0.4.0, maintained, last update Q1 2025) provides a Bevy plugin that exposes an `Engine` with access to Bevy’s `World` and `Commands`. It allows defining custom functions in Rust and calling them from Rhai.  
**Maturity:** Small but functional; used in indie projects. For our platform we may extend it.  
**Idiomatic pattern:** Define a `ScriptPlugin` that registers systems to tick Rhai scripts attached to entities, passing `&World` and `&mut Commands` via a wrapped scope. Scripts can query components and issue commands. *Established practice.*

### B.2 Sandbox Hardening
Rhai’s sandbox is *not* a security boundary against malicious host (modified client), but it protects the runtime from buggy/abusive content.  
**Required limits:**
- `engine.set_max_operations(1_000_000)` – operation budget per script run  
- `engine.set_max_call_levels(64)` – recursion depth  
- `engine.set_max_expr_depths(64, 64)` – expression nesting  
- `engine.set_max_modules(10)` – limit imported modules  
- **Disable** dangerous features: `internals`, `no_std`, `unchecked` – ensure only the whitelisted API is visible.  
- **Whitelisting:** Build the engine with only the functions/types explicitly registered; no `eval` of arbitrary strings from untrusted sources. Scripts are compiled once and only AST is shipped, preventing code injection.  
- **Execution timeout:** Rhai currently lacks a built‑in timeout; workaround: use a `watchdog` that checks `engine.set_progress(...)` callback and throws an error after N operations. *Inferred from documentation.*

### B.3 Hot‑Reload of Rhai Scripts
Scripts are fetched as bytes at runtime, compiled to AST, and stored. On update, the old AST is replaced and scripts are re‑attached to entities. Bevy’s asset system can handle this. Both browser and native identical. *Established practice.*

### B.4 Performance Ceiling
Rhai is a tree‑walking interpreter with bytecode compilation. Measured benchmarks:  
- Simple game logic (e.g., 10,000 updates/second) on modern CPUs.  
- Bottleneck is in heavy loops; the recommendation that hot loops stay in Rust holds.  
- **Native does not accelerate Rhai** – the interpreter speed is CPU‑bound, same in WASM and native.  
**Confidence:** Established practice, many users.

### B.5 Escape Hatch for Perf‑Critical User Logic
Options:  
1. **Rust compiled to WASM components:** advanced authors can write a WASM module that plugs into the game, loaded at runtime (similar to Roblox’s Luau vs. compiled modules). This requires a WASM runtime (e.g., `wasmtime` embedded) with hot‑reload. Complexity is high.  
2. **Native‑only acceleration:** Non‑goal (browser‑first).  
3. **Pre‑compiled Rust systems exposed to scripts:** We ship a library of performance‑sensitive functions (e.g., custom movement systems) that scripts can call.  
**Recommendation:** Provide a “fast path” library of Rust functions callable from Rhai; for authors who outgrow that, allow uploading a WASM module as an optional advanced feature (gated behind a “pro” author tier). Default remains Rhai. *Speculation* – need author demand data.

## C. Transport (WebRTC DataChannels)

### C.1 Rust WebRTC Stack
- **`str0m`** (0.7, Jan 2025) – sans‑IO, synchronous API, focused on DataChannels and media. Used by LiveKit, Cloudflare Calls. Excellent for servers. Supports unreliable/unordered channels via `RTCDataChannel` parameters. Mature.  
- **`webrtc-rs`** (0.11, Feb 2025) – full stack (ICE, DTLS, SCTP) with async support. Heavier resource usage; still maintained.  
**For Mode 3 server (authoritative hub):** `str0m` is preferred due to lower overhead and sans‑IO design that integrates well with Bevy’s ECS.  
**For native peers:** Could use either, but `str0m` also works with a thin I/O driver.  
**WASM/browser:** Uses the browser’s `RTCPeerConnection` API via `web-sys`.  
**Recommendation:** Adopt `str0m` server‑side and for native clients, and implement a common trait (see C.3). *Established practice.*

### C.2 Bevy‑Native WebRTC P2P (matchbox)
`matchbox` (0.9, Dec 2024) provides a socket‑like abstraction over WebRTC DataChannels with a built‑in signaling server (`matchbox_server`). `bevy_matchbox` integrates it into Bevy as a network socket.  
- It uses a **single** DataChannel per peer connection (configurable reliable/unreliable).  
- To support our **two‑channel split** (reliable + unreliable), we would need either to multiplex over one channel (less clean) or modify `matchbox` to open two channels.  
**Evaluation:** The abstraction is ideal for P2P meshes, but the two‑channel requirement is non‑negotiable. Forking `matchbox` to support an additional channel is straightforward; we can contribute back. Alternatively, build a thin wrapper around `web-sys`/`str0m` that yields exactly the needed pair of channels per peer.  
**Recommendation:** Fork `matchbox`, extend to support `(reliable, unreliable)` channel pairs. *Inferred from adjacent evidence.*

### C.3 Browser/Native Asymmetry & Common Abstraction
Define a trait `Transport`:

```rust
#[async_trait]
trait PeerConnection {
    async fn open_unreliable_channel(&mut self) -> Result<Sender<Bytes>, Error>;
    async fn open_reliable_channel(&mut self) -> Result<Sender<Bytes>, Error>;
    // incoming channels are handled via callbacks
}
```

- **Browser implementation:** Uses `web_sys::RtcPeerConnection` and the JS API, mapping events to Rust futures.  
- **Native/Server implementation:** Uses `str0m` (or `webrtc-rs`), providing the same interface.  
**Interop:** Both sides speak standard WebRTC (SDP/ICE), so a browser peer can connect to a native server and vice‑versa. This has been proven by `matchbox`’s browser‑native demos. *Established practice.*

### C.4 Channel Configuration
**Unreliable/Unordered:**
- `ordered: false`, `maxRetransmits: 0` (or `maxPacketLifeTime: 0`). This tells the SCTP layer to deliver messages as they arrive, no retransmission. Last‑write‑wins on the receiver.  
**Reliable/Ordered:**
- `ordered: true`, no retransmit limit (default reliable). Messages arrive in order, retransmitted if lost.  
*Established practice.*

### C.5 Signaling Protocol over WebSocket
A central WebSocket service (see I.1) exchanges JSON/MessagePack messages for SDP offers/answers and ICE candidates. Protocol:
- Client connects → sends `{type: "register", session_id, mode, version}`.  
- For matchmaking, server groups compatible peers and forwards SDP/ICE between them.  
After connection, the WebSocket is used for coordinator bookkeeping (Mode 2) and session list updates. *Established practice.*

### C.6 NAT Traversal (STUN/TURN)
- **STUN-only** fails for ~8–14% of users (symmetric NAT, mobile carriers). Mode 2 *accepts* this: those peers cannot join free P2P sessions; they are notified and can upgrade to Mode 3 (which includes TURN).  
- **TURN** is required for Mode 3 and for the signaling service’s health.  
**coturn** (open source) is battle‑tested. Managed TURN services (Twilio, Xirsys) simplify operation.  
**Bandwidth cost:** TURN relays all traffic. For a typical session of 4 players at 64 kbps (state) + 10 kbps (reliable), total ~300 kbps per session. Cost at $0.01/GB ≈ $0.00004 per hour per session. Negligible.  
**Provisioning:** Spin up a TURN server per region only when a Mode 3 session is active; tear down when idle. Or use a pooled TURN server with authentication tokens tied to the paid session. *Established practice.*

## D. State Replication Protocol (Core Subsystem)

### D.1 Build‑vs‑Reuse Verdict (Pivotal)
**Candidate crates evaluated:**
- **`lightyear`** (0.17, Feb 2025): Client‑server prediction/rollback, supports host‑server (single authority) and can be extended to P2P, but **per‑entity distributed authority** is not a design goal. The ownership model is implicitly client‑server; adapting it to Mode 2 would require significant modification.  
- **`bevy_replicon`** (0.27, Jan 2025): Server‑authoritative entity replication over `renet`. Assumes a central server owning all entities. Bending to per‑peer authority would mean running a “server” on each peer, which wastes resources and complicates the ownership handoff.  
- **`renet`** itself: A transport layer (UDP/WebTransport), not a replication protocol; it does not handle prediction/interpolation, just reliable/unreliable channels. We would still need to build the replication logic.  
**Conclusion:** None of the existing crates directly support the fixed model (distributed per‑entity authority, predict‑own, interpolate‑others, handoff). They would require a fork and heavy customisation. Moreover, we must work over WebRTC DataChannels (not raw UDP), so `renet`’s native UDP transport would need replacement anyway.

**Verdict: Build a custom state replication protocol on top of our WebRTC transport abstraction.**  
This protocol is explicitly designed for the three modes, swapping only authority assignment. We can borrow ideas from `lightyear`’s prediction/interpolation and `bevy_replicon`’s component replication, but the core is purpose‑built. *Inferred from analysis of crate designs; confidence medium-high.*  
This approach avoids fighting framework assumptions, keeps dependencies minimal, and yields a lean implementation tuned for our exact bandwidth/ownership model.

### D.2 Wire Format
- **Snapshot encoding:** For each owned entity, we send a compact binary blob: `entity_id` (u32), `generation` (u32), followed by a component mask and serialised component data.  
- **Serialization:** Use `postcard` (no_std, WASM‑friendly, zero‑copy decoding) or `bincode` with a custom configuration. Postcard is preferred for its small code size and good performance.  
- **Delta encoding:** For bandwidth savings, send only changed components per tick (dirty tracking via Bevy change detection). The initial sync sends a full snapshot. Delta format: `(component_id, data)` pairs.  
- **Quantization:** Positions can be quantised to f32 with fixed‑point or simply sent as f32; bandwidth is not extremely tight. For further savings, custom `Serde` impls can compress rotations (quaternion quantisation).  
- **Bandwidth per entity:** Position (12 bytes), rotation (16 bytes half‑float), velocity (12 bytes) = 40 bytes. At 20 Hz, 800 bytes/s per entity. 50 entities = 40 KB/s per peer upstream (unreliable). Measured by simple calculation; real‑world overhead adds ~20% for headers and entity IDs. *Inferred.*

### D.3 Prediction & Interpolation
- **Client‑side prediction (for owned entities):** The local peer applies its inputs immediately. When a server/remote correction arrives (Mode 3), the client reconciles by re‑applying inputs from the last acknowledged state.  
- **Entity interpolation (for remote entities):** Buffer snapshots for a time window (e.g., 100 ms) and interpolate between the two most recent states. This hides jitter and drops.  
- **Network tick rate:** 20 Hz (every 50 ms). Interpolation delay ~100 ms (2 ticks). Display rate 60 fps via interpolation.  
- **Bandwidth at target:** As computed above; 100 entities @ 20 Hz ≈ 80 KB/s upstream for a full mesh with 4 peers. Each peer sends its own 25 entities ≈ 20 KB/s. Acceptable. *Established practice in networked games.*

### D.4 Interest Management (AOI)
- Spatial grid (e.g., `bevy_spatial` / `kdbush`) mapping entity positions to grid cells. Each peer subscribes to updates for entities within a radius (e.g., 100 m). Out‑of‑view state is **not** sent.  
- In Mode 3, this also serves as read‑cheat mitigation: the server withholds positions of entities the player cannot see.  
- Implementation: When a peer’s owned entity moves, the server/grid determines which other peers need an update; only they receive it. *Established pattern; inferred for Bevy.*

### D.5 Ownership Transfer (Mode 2 Sharp Edge)
**Protocol:**
1. Current owner A sends a `HandoffRequest(entity_id, target_peer)` to the coordinator via the reliable channel.  
2. Coordinator verifies A is the current owner (checks its bookkeeping). If yes, it sends `HandoffAssign(entity_id, new_owner=target)` to all peers in the session.  
3. Upon receiving `HandoffAssign`, all peers update their authority table: the entity is now owned by target. The former owner stops sending updates; the new owner starts sending.  
4. If the target peer is unreachable or drops during handoff, the coordinator times out (e.g., 2 s) and reassigns to the oldest surviving peer (host migration logic).  
**Contested claims:** If two peers claim ownership of the same entity simultaneously, the coordinator rejects all claims and assigns ownership based on a deterministic rule (lowest peer ID), then broadcasts. This prevents split‑brain.  
**Failure analysis:**  
- A drops before completing handoff → no ack, coordinator assigns to target directly (if target alive) or back to A’s successor.  
- Target drops after coordinator’s assign → the entity becomes orphaned until anti‑entropy detects missing updates and the coordinator reassigns.  
**Latency:** One reliable round‑trip to coordinator (≈20–50 ms). This is visible but acceptable for discrete actions (throwing a ball). *Inferred from consensus protocols, needs testing.*

### D.6 Cross‑Owner Interaction (Quality Gap)
Example: Ball (owned by A) hits target (owned by B).  
- A simulates the ball’s movement (its own entity) and may decide it collides with B’s entity based on B’s last known position. B, as owner of its own player, may have moved away. Result: visual pop on B’s screen when the ball “hits” where the player used to be.  
- This jank is inherent because no single authority exists; it cannot be eliminated without a central server.  
- **Bound:** The error is limited to the prediction time (latency) and the relative velocity. For casual games, it’s acceptable. The difference defines Mode 2 vs Mode 3 quality.  
**Quantification:** At 50 ms latency and a relative speed of 10 m/s, the spatial error is 0.5 m. Noticeable but not game‑breaking for non‑competitive play. *Inferred, needs experiment.*

### D.7 Anti‑Entropy / Resync without CRDT
- Every 1 second, each peer broadcasts a `StateHashList` containing `(entity_id, hash(state), version)` for all entities it owns.  
- Receivers compare with their local version; on mismatch, they request a full snapshot from the owning peer via the reliable channel.  
- This replaces CRDT eventual consistency and guarantees that after a disruption, state converges without conflicts (single‑owner). *Established pattern.*

## E. Anti‑Cheat & Verification

### E.1 Mode 2 Plausibility/Bounds Validation
Checks performed by receiving peers on incoming state (for entities they don’t own):
- Speed cap (e.g., player cannot move > 10 m/s)  
- Position delta vs. elapsed time (teleport detection)  
- Acceleration/jerk limits  
- Rate‑limiting of certain actions  
These are simple numeric checks in the replication receive system. *Established practice.*

### E.2 Signed Ops (Tamper‑Evidence)
- **Keypair:** `ed25519-dalek` (pure Rust, WASM compatible). Signing 64‑byte message ≈ 50 µs; verification ≈ 100 µs.  
- At 20 Hz, signing every unreliable message per entity would be costly. Instead, sign **only reliable channel messages** (ownership handoffs, join/leave, anti‑entropy hashes) and a periodic “heartbeat” that includes a hash of recent unreliable state. This provides tamper evidence: if a peer modifies its position and sends fake state, the signed periodic summary will reveal the cheat (post‑session analysis) or another peer can detect the hash mismatch when receiving the summary.  
- **Per‑message signing cost:** For 10 reliable messages per second, signing overhead <1 ms — negligible. *Established practice, Dalek benchmarks cited.*

### E.3 Mode 3 Server‑Authoritative Validation
The headless server runs the full simulation (including physics) and validates all client inputs:
- Inputs are applied, but the server‑side result becomes the authoritative state. Client prediction is corrected.  
- Interest management prevents a hacked client from reading entity positions outside its view (read‑cheat mitigation).  
- Because the server is under our control, we can also run server‑side heuristics (e.g., aimbot detection) not possible in Mode 2. *Established practice.*

### E.4 Trust Model per Mode (Security Synthesis ties to L)
- **Mode 1:** All local, no trust needed.  
- **Mode 2:** Trusts that peers run the unmodified client? No — client can be modified. Signed ops provide *tamper evidence* but not prevention. Plausibility checks catch extreme cheats. The real guarantee is **social** (friends‑only sessions) and **opt‑in**. Players who want fair competition choose Mode 3.  
- **Mode 3:** Server is the trusted authority. Cheats are limited to client‑side assists (aimbot, wallhack via rendering) because the server validates game state. Read‑cheats are partially mitigated by interest management. The browser attestation ceiling means we cannot prevent client modification, but server authority renders most cheats ineffective. *Synthesis of Section 2.11.*

## F. Identity, Accounts, Billing, Persistence

### F.1 Device Keypair
- **Generation:** In WASM, use `getrandom` with browser’s Crypto API to seed `ed25519-dalek` key generation. Store the private key in IndexedDB (encrypted with a local PIN if desired).  
- **WebCrypto non‑extractable keys** are problematic because they can’t be used for arbitrary signing; we’d have to use SubtleCrypto sign, but that locks us to browser‑specific APIs and complicates native. Better to generate in WASM and store.  
- **Native:** Use OS keyring (`keyring` crate) or a simple file in user data.  
- **Survivability:** IndexedDB is durable across page reloads but can be cleared by user. Losing the keypair means losing account access (if associated with an account). For anonymous free tier, this is acceptable; for paid account, we bind the keypair to the email‑based account and allow re‑provisioning. *Inferred from web practices.*

### F.2 Full‑Server Auth & Billing
- **Auth:** Email + password (hashed) via `argon2`, or OAuth (Google, GitHub). Use `axum` + `sqlx` for auth endpoints.  
- **Billing:** **Stripe** (subscriptions with webhooks). The backend stores `stripe_customer_id` and subscription status in Postgres. When a Mode 3 session is requested, the matchmaker verifies active subscription.  
- **Security:** Card data never touches our server; Stripe.js tokenises on the frontend. *Established practice.*

### F.3 Postgres Schema & Object Storage
**Postgres tables:**
- `users` (id, email, password_hash, created_at)  
- `sessions` (id, game_version, mode, start_time, end_time, participants[], server_id)  
- `subscriptions` (user_id, stripe_sub_id, status, current_period_end)  
- `published_games` (content_id, author_id, version_engine, version_content, version_schema, bundle_path, created_at, moderation_status)  
- `rankings` (user_id, game_id, score, match_id)  

**Object storage:** S3‑compatible (MinIO self‑hosted or AWS S3). Game bundles and saved snapshots are stored by content hash (BLAKE3) in a bucket. *Established practice.*

## G. Publish Pipeline & Versioning

### G.1 Content‑Addressed IDs & Bundle Format
- Each published game is packaged as a `.tar.gz` or `.zip` containing `manifest.json` and asset files.  
- The content ID = `blake3` hash of the entire bundle.  
- The manifest includes: engine min version, content version (semver), schema version (for migration), list of scripts/assets with their individual hashes.  
*Established practice (IPFS-like).*

### G.2 Version Compatibility & Matchmaker Enforcement
- **Engine:** Matchmaker allows joining only if client’s engine version >= game’s declared minimum.  
- **Content & schema:** Must match exactly (identical hash). This guarantees deterministic behaviour; no cross‑version desync.  
- **Schema migration:** When an author updates, they bump the content version; old sessions stay on the old version. Authors can provide migration scripts (Rhai) to convert saved state. *Established practice.*

### G.3 UGC Moderation Tooling
- **Automated scanning:**  
  - **Illegal content:** PhotoDNA-like hashing for CSAM (via services like Thorn) – mandatory.  
  - **Models/textures:** Run virus scans, check for malicious polyglots. Use `clamav` or similar.  
  - **Audio:** Content ID matching for copyrighted music (Shazam API).  
- **Human review queue:** Flag reported assets; review interface shows model previews.  
- **Realistically automatable:** Copyright detection, known illegal hashes, file‑type validation. Human review necessary for contextual violations (e.g., hate symbols). *Established practice at scale (Roblox, Minecraft).*

## H. Hot Update & Distribution

### H.1 Content Hot‑Reload Mechanics
- Browser: `fetch` bundle from CDN (cache‑busted via content ID), unzip in WASM (using `zip` crate), feed new Rhai scripts and assets to Bevy’s `AssetServer`. Old assets are evicted.  
- Native: Same, but using local HTTP or file system.  
*Established practice with Bevy’s asset hot‑reloading.*

### H.2 Engine Updates
- **Browser WASM:** Use a service worker to cache the new WASM module under a versioned URL. When a new version is detected (polling the service), prompt user to reload. Or simply reload the page with new URL parameter, forcing cache miss.  
- **Native auto‑update:** Use `self_update` crate (backed by GitHub Releases) to download and replace the binary, then relaunch. Windows requires code signing; macOS notarisation. Distributing plain executables (not webview) is standard for Bevy. *Established practice.*

### H.3 Version‑Gating on Session Join (Desync Defense)
As described in D.1/G.2, the matchmaker prevents cross‑version play. This is the only line of defense against desyncs caused by engine differences. *Inferred.*

## I. Session / Matchmaking / Coordination Service

### I.1 Central WebSocket Service
Build a single Rust service (e.g., `axum` + `tokio-tungstenite`) that handles:
- WebSocket connections for signaling, session registry, matchmaking.  
- HTTP endpoints for game listing, version check, billing webhooks.  
- Stores active sessions in Redis for horizontal scalability.  
**Message protocol:** JSON over WebSocket for ease of debugging; switch to MessagePack if bandwidth becomes an issue.  
*Established practice.*

### I.2 Coordinator Election & Host Migration (Mode 2)
- On session creation, the first peer is the coordinator.  
- If coordinator disconnects, the oldest surviving peer (by join order) takes over. It sends a `CoordinatorChange` to all peers via reliable channel.  
- The new coordinator takes over the bookkeeping (peer list, handoff arbitration). No state transfer needed because game state is replicated across peers. *Established practice in P2P games.*

### I.3 Scaling
- The service is stateless except for active WebSocket connections and session metadata in Redis.  
- Use Redis Pub/Sub to synchronise session registry across multiple service instances.  
- Horizontal scaling: run N instances behind a load balancer, sticky sessions for WebSocket. *Established practice.*

## J. Mode 3 Server Orchestration & Infrastructure

### J.1 Per‑Session Authoritative Sim Processes
- Each competitive session spawns a headless Bevy binary (stripped of rendering) that loads the specific game content bundle. It runs the Rhai logic as the central authority.  
- **Container‑per‑session:** Wrap the binary in a minimal Docker image (Alpine + binary + assets). Use Kubernetes with a custom controller or **Agones** (game server orchestration).  
- **Cost model:** A small VM (2 vCPU, 2 GB RAM) can run ~5-10 concurrent headless sessions (depending on simulation complexity). At $0.05/hour per VM, cost per session ≈ $0.005–0.01/hour. With a subscription price of $5/month, even 20 hours of play per user yields positive margin. *Inferred from cloud costs; needs load testing.*

### J.2 Orchestration & Handoff
- Matchmaker receives Mode 3 request → checks subscription → requests an available server from the fleet manager.  
- Fleet manager provisions a new container (or picks an idle one), passes it the session ID, game bundle URL, and connection details.  
- Server connects back to the signaling service, which forwards SDP to peers. Peers connect to the server’s WebRTC endpoint.  
- Tear down after session ends (no players for 5 min). *Established game server pattern.*

### J.3 TURN Provisioning
- When a Mode 3 session starts, the orchestration spins up a coturn instance (or allocates from a warm pool) and provides credentials to the server. Peers who cannot directly connect (symmetric NAT) are instructed to use TURN.  
- Bandwidth cost: as calculated earlier, negligible. Use managed TURN to avoid self‑hosting complexity. *Established practice.*

## K. Cross‑Cutting WASM / Browser Constraints

### K.1 Threads: Go/No‑Go
**Recommendation: No cross‑origin isolation, no threads.**  
- The complexity and restrictions (can’t be embedded, breaks many CDN configurations, requires site‑wide header changes) are not worth the performance gain for our target audience.  
- Single‑thread Bevy is sufficient for UGC games that are typically not CPU‑bound (Rhai scripts are interpreter‑limited anyway).  
**If later needed,** offer an opt‑in separate build with threads enabled for advanced games that are hosted on dedicated pages with controlled headers. *Compromise ledger entry.*

### K.2 Memory (4 GB Ceiling)
- The `wasm32` 4 GB limit is sufficient for client‑side games. UGC worlds are unlikely to exceed that in the browser.  
- **Memory64** is not yet stable in Rust/WASM; ignore for now. *Established constraint.*

### K.3 Persistent Browser Storage
- **IndexedDB:** Used for device keypair and cached assets. Quota is typically > 10% of disk space (navigator.storage.estimate). Eviction is rare but can happen under storage pressure.  
- Mitigation: For keypair, allow re‑provisioning via account login; for assets, they can be re‑downloaded. *Established web storage practice.*

### K.4 Cold‑Load Budget Synthesis
Target: < 5 seconds on 10 Mbps.  
- Binary (optimised): 2.0 MB  
- Essential assets (UI, initial Rhai scripts): 1.0 MB  
- Total: 3.0 MB → ~2.4 s load time.  
Techniques: lazy‑loading of game assets, `wasm-opt`, CDN with Brotli compression (WASM binary compresses to ~500 KB). *Inferred from A.3 and H.1.*

## 2. Build‑vs‑Reuse Verdict for State Replication (D.1)

**Verdict:** Build a custom state replication protocol on top of WebRTC DataChannels.  
**Justification:**  
- Existing Bevy replication crates (`lightyear`, `bevy_replicon`) are architecturally coupled to a client‑server or single‑authority model. Per‑entity distributed authority (Mode 2) and seamless handoff are not supported.  
- `renet` is a transport, not a replication layer, and would need a WebRTC backend anyway.  
- Building a thin, tailored protocol lets us enforce the fixed constraints directly, keep dependencies low, and avoid fighting framework abstractions.  
- The protocol is well‑defined: per‑entity snapshots/deltas, predict‑own/interpolate‑others, ownership handoff via coordinator, anti‑entropy via state hashes. This is a bounded amount of work.  
**Confidence:** High (design is inferred from established multiplayer patterns; implementation risk exists but is manageable).

## 3. Architecture Diagrams (Text Description)

Three diagrams: overall system components, data flow per mode, and trust boundaries.

### Overall System Components
```
[Browser Client (WASM)]  <--WebRTC--> [Other Browser Clients]
       | (WebSocket)
[Signaling/Matchmaking Service] -- (HTTP/Redis) -- [Platform DB/Storage]
       |
[Native Client] <--WebRTC--> [Browser Client] or [Mode 3 Headless Server]
       |
[Mode 3 Server Fleet] -- managed by Orchestrator
       |
[TURN Server (paid sessions only)]
```

### Data Flow – Mode 2 (Hybrid P2P)
- Peers exchange unreliable state updates directly (mesh) for entities they own.  
- Reliable channel to coordinator for session bookkeeping, handoff arbitration, and anti‑entropy.  
- Trust boundary: peers are untrusted; signed ops on reliable channel; bounds checks on incoming state.

### Data Flow – Mode 3 (Full Server)
- Clients send inputs to headless server via unreliable channel.  
- Server simulates, sends authoritative state to each client via unreliable, and critical events via reliable.  
- Server enforces interest management; client prediction with server reconciliation.  
- Trust boundary: server is trusted.

### Trust Boundaries Marked
- Between client and server (server authoritative).  
- Between peers (untrusted).  
- Between client and signaling service (authenticated but not authoritative over game state).  
- At publish pipeline: only point where UGC is scanned; after that, content runs in the client with sandbox.

## 6. Phased Build Sequence (Vertical‑Slice‑First)

**Phase 1 – Core Replication & Mode 2/3 Proof (8–12 weeks)**
- Implement the WebRTC transport abstraction for browser (via `matchbox` fork) and native (`str0m`).

## 7. Compromise Ledger (Browser vs Native)

| Limitation | Quantified Cost | Stance |
|------------|-----------------|--------|
| **Single‑threaded WASM** | No parallel ECS; complex games may drop frames. 4‑core native could be 3‑4x faster in simulation. | Accept; target casual UGC. Provide optional threaded build for advanced creators. |
| **No kernel anti‑cheat** | Mode 3 only server‑side validation; aimbots/wallhacks possible via rendering. | Accept; maximum achievable in‑browser. Marketing emphasizes “fair‑enough competitive”. |
| **Memory limit 4 GB** | Large worlds may exhaust memory faster than native. | Accept; wasm64 not ready. Compress assets, streaming. |
| **Cold‑load time 2–4 s** | Slower than native instant start. | Accept; standard web app expectation. |
| **IndexedDB keypair fragility** | Losing private key = account loss if not backed up. | Mitigate by binding to email account; provide key export. |
| **WebRTC DataChannel overhead** | Higher latency than raw UDP (5–20 ms extra). | Accept; necessary for browser P2P. |
| **STUN‑only P2P connectivity failures** | ~10% of users cannot join free sessions. | Accept; upsell to Mode 3 with TURN. |
| **UGC moderation only at publish** | Cannot scan content at runtime; malicious asset could be loaded after publish. | Rely on sandbox; content‑addressed, no on‑the‑fly modification allowed. |

All compromises are documented, and the platform’s design inherently bounds their impact.:wq
